Architecture

LumenRL follows a controller/worker split: a Ray driver owns scheduling and batching, while specialized workers host ATOM inference, policy training, reference models, rewards, and critics. All cross-worker payloads are DataProto batches.

Ray Controller

The controller (lumenrl/controller/) connects Ray clusters, builds worker groups, and dispatches DataProto batches. It handles: bootstrapping Ray, grouping GPUs into actor/rollout/ref/reward/critic pools, and sequencing the RL outer loop: generate → score → ref/KL → train.

Workers

WorkerResponsibility
RolloutATOM-backed generation, logits/log-prob capture, FP8 rollout quant hooks
ActorPolicy forward/backward under FSDP2 or Megatron
ReferenceFrozen reference policy for KL-regularized algorithms
RewardRule or model-based scoring surfaced as rewards tensors
CriticValue heads for PPO-style advantage estimation
HybridColocated paths when memory or placement favors fused processes

DataProto Protocol

DataProto is the universal batch container: a dict[str, torch.Tensor] plus meta for non-tensor fields. The controller merges partial batches from DP workers and splits for micro-batching inside actor steps.

Engine Layer

  • ATOM inference — wraps generation, KV-cache extensions, and weight loaders so rollout matches training naming and sharding.
  • Lumen training — selects FSDP2 or Megatron, applies Lumen quantization hooks, and applies MoE R3 replay during forward passes.

Quantization Stack

Rollout quantization uses FP8RolloutQuantizer and FP8KVCacheQuantizer; training uses FP8TrainingManager. When FP8 rollouts diverge from BF16, apply_rollout_correction adjusts advantages using TIS or MIS.

MoE R3

R3Manager coordinates RouterRecorder (inference) and RouterReplayer (training), with tensors carried via DataProto.add_router_distributions. This closes the router mismatch that otherwise spikes policy–inference KL.

DAPO Training Example

Decoupled Advantage Policy Optimization with asymmetric clipping and dynamic sampling.

When to Use DAPO

  • Asymmetric clipping — different bounds for ratio increase vs decrease
  • High-variance rewards — dynamic sampling auto-filters degenerate groups
  • Overlong responses — overlong reward shaping penalizes excess length

BF16 Baseline

  name: dapo
  dapo:
    num_generations: 4
    clip_ratio_low: 0.8
    clip_ratio_high: 1.28
    kl_coeff: 0.02
    dynamic_sampling: true
    overlong_reward_shaping: true
    token_level_pg: true

FP8 DAPO

python examples/run_dapo.py --config configs/dapo_dense_fp8.yaml

  rollout:
    precision: fp8
  training:
    fp8: hybrid

    enabled: true

FP8 Quantization

LumenRL implements an FP8-RL style stack: low-precision rollouts for throughput, hybrid FP8 training in Lumen, and importance-sampling correction.

FP8 Rollout (ATOM Path)

W8A8 Linear Layers

FP8RolloutQuantizer replaces eligible nn.Linear modules with blockwise W8A8. Weights are quantized once per replacement; activations use dynamic scales from 128-wide blocks.

FieldMeaning
precision"fp8" enables rollout quantizers; "bf16" disables
use_deep_gemmPrefer fused GEMM when deep_gemm is installed
num_first/last_layers_in_bf16Keep boundary layers in BF16 for stability

KV-Cache Quantization

FP8KVCacheQuantizer maintains per-projection FP8 scales on q_proj, k_proj, v_proj. recalibrate_scales should run whenever policy weights change.

Weight Quantizer

WeightQuantizer provides standalone blockwise FP8 packing for state_dict tensors.

FP8 Training (Lumen Path)

FP8TrainingManager reads config and calls lumen.quantize.enable:

YAML FieldEffect
quantization.training.fp8When set (e.g. "hybrid"), overrides rollout precision
quantization.training.fp8_recipe"blockwise" (default) or "tensorwise"
quantization.training.fp8_weight_cacheRegisters optimizer post-step hooks to avoid re-quantization

Rollout Correction (TIS and MIS)

TIS (Truncated Importance Sampling)

Let ρt = exp(log πbf16 - log πfp8). TIS clamps ρt to [1/c, c] where c is clip (default 1.5), then multiplies advantages:

from lumenrl.quantization.rollout_correction import token_level_tis
adv_corrected = token_level_tis(bf16_logprobs, fp8_logprobs, advantages, clip=1.5)

MIS (Mean Importance Sampling)

MIS keeps raw ratios but normalizes to mean 1 across tokens:

from lumenrl.quantization.rollout_correction import token_level_mis
adv_corrected = token_level_mis(bf16_logprobs, fp8_logprobs, advantages)

MoE Rollout Routing Replay (R3)

MoE RL couples two different code paths: high-throughput inference routers (ATOM) and training routers (Megatron/Lumen). Small differences yield different expert assignments, which shows up as unstable KL and collapsed expert usage.

Problem Statement

  • Inference stacks may fuse softmax, top-k, and dispatch differently than training
  • FP8 rollouts change logits slightly, altering argmax routes even when BF16 is "close"
  • Expert parallel sharding reorders reductions, amplifying bitwise drift

The result is router inconsistency: the policy gradient is computed against a different routing process than what produced the data.

R3 Solution (Record / Transfer / Replay)

  1. Record — During ATOM rollout, RouterRecorder installs hooks and stores CPU float copies of router logits keyed by layer index
  2. Transfer — R3Manager.transfer_distributions copies recorded tensors into DataProto
  3. Replay — RouterReplayer installs hooks during training to overwrite router outputs with recorded logits

Configuration

moe:
  r3:
    enabled: true
    record_router_logits: true
    replay_mode: distribution   # or hard_assignment
FieldPurpose
enabledMaster switch for recorder/replayer
record_router_logitsWhen false, record phase becomes a no-op
replay_modedistribution feeds soft logits; hard_assignment locks discrete decisions

Monitoring

  • Policy–reference KL — spikes often precede MoE collapse
  • Expert utilization — compute from router_logits with check_expert_utilization
  • Router entropy — compute_router_entropy tracks routing decisiveness over time

Best Practices

  • Enable R3 for all MoE RL runs unless explicitly isolating a baseline
  • Keep record_router_logits=true until you verify hooks are lightweight
  • When debugging, log histograms of router_dist_layer_* norms per layer
  • Pair R3 with FP8 correction when rollouts are quantized

Distributed Training

LumenRL scales along two axes: training backend parallelism (FSDP2 vs Megatron-Core) and cluster orchestration (Ray, multi-node SLURM).

FSDP2 Backend

Megatron-Core Backend

Algorithms

Three policy-gradient trainers—GRPO, DAPO, and PPO—sharing the same DataProto batching and loss helpers.

GRPO

Group Relative Policy Optimization: normalizes scalar rewards within each prompt's num_generations group. Uses policy_gradient_loss with symmetric clip.

  • Group-relative advantages — reshape to [B//G, G], subtract group means, divide by group std
  • Clipped surrogate — uses clip_ratio

DAPO

Decoupled Advantage Policy Optimization adds stabilizers on top of GRPO-style advantages:

  • Asymmetric clip — ratios clamp to [clip_ratio_low, clip_ratio_high]
  • Dynamic sampling — groups with near-zero reward variance are masked out
  • Overlong shaping — optional linear penalty for responses exceeding max length

PPO

Classic Proximal Policy Optimization with critic network and GAE-Lambda:

  • GAE-Lambda — constructs token-level advantages and returns
  • Value loss — clips value deltas similar to policy ratio clip
  • Entropy bonus — optional exploration encouragement

Loss Functions Reference

FunctionRole
policy_gradient_lossSymmetric clipped policy surrogate
asymmetric_clip_lossDAPO-style asymmetric clipping
value_lossClipped critic regression
kl_penaltyReference KL penalty
entropy_bonusSampling entropy surrogate

Choosing an Algorithm

ScenarioSuggested
Preference-style scalar rewards, no criticGRPO
Long responses, asymmetric off-policy noiseDAPO
Classic actor–critic with value bootstrappingPPO
Warning: GRPO and DAPO assume contiguous blocks of size num_generations. If your sampler interleaves prompts, normalize ordering before calling compute_advantages.

DataProto API

lumenrl.core.protocol.DataProto is the tensor batch exchanged between Ray workers. Tensors live on CPU for serialization; each worker calls .cuda() before math.

Algorithms API

lumenrl.algorithms implements policy-gradient algorithms on DataProto batches. All concrete classes inherit from BaseAlgorithm.

BaseAlgorithm

class BaseAlgorithm:
    def __init__(self, config: LumenRLConfig) -> None: ...
    def compute_advantages(self, batch: DataProto) -> DataProto: ...
    def compute_loss(self, batch: DataProto) -> tuple[Tensor, dict[str, Any]]: ...
    def get_config(self) -> dict[str, Any]: ...

GRPOAlgorithm

Requires rewards shaped [B]. Builds advantages by normalizing within groups of num_generations.

from lumenrl.algorithms import GRPOAlgorithm
from lumenrl.core.config import LumenRLConfig

algo = GRPOAlgorithm(LumenRLConfig())
batch = algo.compute_advantages(batch)
loss, metrics = algo.compute_loss(batch)

DAPOAlgorithm

Extends GRPO with optional dapo_sample_mask, overlong reward shaping, and asymmetric objectives.

PPOAlgorithm

Requires values [B, T], writes advantages and returns. Adds 0.5 * value_loss to policy loss.

ALGORITHM_REGISTRY

from lumenrl.core.registry import ALGORITHM_REGISTRY
cls = ALGORITHM_REGISTRY.get("grpo")
algo = cls(config)

Loss Functions

policy_gradient_loss

from lumenrl.algorithms.loss_functions import policy_gradient_loss
loss_pg = policy_gradient_loss(log_probs, old_log_probs, adv, clip_ratio=0.2, mask=mask)

asymmetric_clip_loss

Same as above but clamps ratios to [clip_low, clip_high] (DAPO).

value_loss

Clipped value regression combining clipped/unclipped MSE terms.

kl_penalty

Token-average ref_logprobs - logprobs, used as approximate KL penalty.

entropy_bonus

Surrogate entropy from sampled log-probs; subtract coeff * entropy_bonus from loss.

Rollout Correction

Truncated importance sampling: build exp(log_bf16 - log_fp8), clamp to [1/clip, clip], multiply advantages.

Mean-normalized ratios so average weight is 1.
