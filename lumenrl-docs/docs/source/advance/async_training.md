# Fully Async Training

LumenRL supports a **fully-async training mode** that decouples rollout (generation) from policy training, inspired by [VERL's fully_async_policy](https://github.com/verl-project/verl/blob/main/docs/advance/fully_async.md) and techniques from AReaL, Magistral, StreamRL, and AsyncFlow.

In the default *colocate* mode, rollout and training share the same GPUs and take turns — the GPU is idle during the other phase. In async mode, a background **Rollouter** thread continuously generates samples into a **MessageQueue** while the **Trainer** consumes batches for gradient updates. Periodic **ParameterSync** pushes updated weights from trainer to rollouter.

## Architecture

![LumenRL Fully Async Training Architecture](/_static/async_architecture.svg)

1. **Rollouter** generates sequences one sample at a time and pushes them into the MessageQueue, with production speed controlled by staleness.
2. **MessageQueue** buffers samples between Rollouter and Trainer with freshness control.
3. **Trainer** fetches `require_batches × ppo_mini_batch_size` samples at a time. After `trigger_parameter_sync_step` local updates, it pushes new weights to the Rollouter.
4. **WeightSync** implements filesystem-based parameter synchronization (safetensors snapshots).

## Supported Modes

| Mode | Settings | Description |
| --- | --- | --- |
| **On-policy** | `staleness=0, sync_step=1` | Rollouter produces one batch, Trainer trains and syncs immediately. |
| **Stream off-policy** | `staleness=0, sync_step>1` | Synchronous streaming — multiple local updates between syncs. |
| **Async + stale** | `staleness>0, partial=False` | Allows stale samples up to threshold. Rollouter waits for active tasks on sync. |
| **Async + partial** | `staleness>0, partial=True` | Rollouter interrupts active generation on sync, saves progress for next round. |

## Configuration

Add the `async_training` section to your YAML config:

```yaml
async_training:
  enabled: true
  require_batches: 4          # mini-batches per training step
  trigger_parameter_sync_step: 4  # local updates before weight sync
  staleness_threshold: 0.5    # max fraction of stale samples (0 = sync)
  partial_rollout: true       # interrupt rollout on sync
  use_rollout_log_probs: true # compute old_log_probs during rollout
  queue_maxsize: 64           # sample buffer size
  weight_sync_dir: /tmp/lumenrl_weight_sync
```

### Parameter Reference

| Parameter | Default | Description |
| --- | --- | --- |
| `enabled` | `false` | Enable async training mode |
| `require_batches` | `4` | Number of mini-batch-sized chunks fetched per training step |
| `trigger_parameter_sync_step` | `4` | Local gradient updates between weight syncs |
| `staleness_threshold` | `0.0` | Max stale sample fraction (0 = synchronous, 0.5 = up to 50% stale) |
| `partial_rollout` | `false` | Interrupt in-progress generation on weight sync |
| `use_rollout_log_probs` | `true` | Compute `old_log_probs` in rollout phase (recommended for async) |
| `queue_maxsize` | `64` | Maximum samples buffered in the message queue |
| `weight_sync_dir` | `/tmp/lumenrl_weight_sync` | Directory for weight snapshot files |

### Staleness Control

- `staleness_threshold=0`: Fully synchronous — the trainer waits for fresh samples.
- `staleness_threshold>0`: Allows samples generated by older parameter versions.  Setting to 0.5 means up to 50% of a training batch can be stale.
- When rollout is fast, `staleness_threshold=1.0` is roughly equivalent to one-step off-policy.

### Parameter Sync Frequency

Between two weight syncs, the trainer processes:

```
trigger_parameter_sync_step × require_batches × ppo_mini_batch_size
```

samples. To compare fairly with colocate mode where `train_batch_size=512`:

```
trigger_parameter_sync_step = train_batch_size / (require_batches × ppo_mini_batch_size)
```

## Quick Start

```yaml
# configs/1a_bf16_async.yaml
policy:
  model_name: Qwen/Qwen3-8B
  generation_backend: atom
  max_response_length: 7168

algorithm:
  name: dapo
  dapo:
    num_generations: 16
    kl_coeff: 0.0

async_training:
  enabled: true
  require_batches: 4
  trigger_parameter_sync_step: 4
  staleness_threshold: 0.5
  partial_rollout: true

num_training_steps: 275
```

Launch:

```bash
torchrun --nproc_per_node=8 -m lumenrl.trainer.main \
  --config configs/1a_bf16_async.yaml
```

## Key Metrics

| Metric | Description |
| --- | --- |
| `async/trainer_idle_s` | Cumulative seconds the trainer waited for samples |
| `async/rollout_idle_s` | Cumulative seconds the rollouter waited for weight sync |
| `async/param_version` | Current parameter version after syncs |
| `queue/size` | Current number of buffered samples |
| `queue/stale_ratio` | Fraction of consumed samples that were stale |

## Tuning Recommendations

1. **Resource balance**: Monitor `trainer_idle_s` and `rollout_idle_s`. If the trainer idles, the rollouter is too slow — consider reducing generation length or using FP8 rollout for faster inference.

2. **Staleness**: Start with `staleness_threshold=0` (sync streaming) and increase to 0.3–0.5 if the rollouter consistently underfeeds the trainer. Values above 1.0 are not recommended.

3. **Require batches**: Higher values (4–8) give better GPU utilization during training but may affect sample ordering. Start with 4.

4. **Sync frequency**: `trigger_parameter_sync_step=4` is a good default. Lower values keep the policy fresher but increase sync overhead. Higher values improve throughput at the cost of more off-policy training.

## Comparison with Colocate Mode

| Aspect | Colocate (default) | Async |
| --- | --- | --- |
| GPU sharing | Shared — swap memory via optimizer offload | Shared or separate |
| Pipeline bubble | Rollout and training are sequential | Overlapped |
| Memory efficiency | Must offload optimizer during rollout | No offload needed (separated) |
| Staleness | Always on-policy | Configurable |
| Throughput | 1× baseline | 1.6–2.7× (depends on config) |
| Complexity | Simple | More tuning knobs |

## ROCm Notes

On AMD MI300X/MI350X GPUs:

- ATOM uses subprocess isolation for inference to avoid NCCL conflicts with the training process group.
- Weight sync uses filesystem (safetensors) rather than NCCL because the vLLM subprocess does not share the trainer's NCCL group.
- `partial_rollout=True` triggers ATOM subprocess termination and restart on each sync, which is the only reliable way to fully free GPU memory on ROCm (see {doc}`/advance/distributed` for background).
